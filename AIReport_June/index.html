01 / 遊戲企劃

讓規格書跟著程式一起活著

這個 AI 助手主要解決過去規格文件「分散、版本混亂、更新不連續」的問題。它會閱讀 Cocos 專案程式碼，協助反向整理成規格文件，並把既有文件收斂成可持續維護的版本。

原本的痛點

  • 有些規格書沒有持續迭代更新。
  • 有更新時常變成另一份新文件。
  • 文件分散，難以確認哪一份才是最新版。

AI 協助方式

  • 讀取 Cocos 專案程式碼，整理成 Markdown 規格。
  • 產出技術版與非技術版，分別給工程師與企劃 / PM 使用。
  • 當程式碼改動時，比對既有規格書並直接更新同一份文件。
目標：規格書不是越長越多份，而是能跟著程式一起維護。

目前進度與下一步

目前已完成「code → 規格（Markdown）」的流程。下一步希望完成「規格（Markdown） → code」，並透過反覆迭代，建立技術文件的框架與規範。

02 / 遊戲品檢

更早踩雷，也更容易重現問題

這個 AI 助手偏向 QA 與自動化測試，核心任務是根據規格書與程式碼產生測試流程，協助團隊更早發現 bug，並在出問題時留下可追查的紀錄。

主要能力

  • 根據規格書與程式碼產生自動化測試腳本。
  • 自動生成 mock data，測試邊界值與極端情況。
  • 模擬玩家行為，執行實際流程測試。

問題紀錄

  • 錯在哪裡。
  • 如何觸發。
  • 如何重現。
對團隊的價值：更早踩雷，出事時也比較好查。

目前進度與下一步

目前比較接近 client 端單元測試，會自行寫假資料做極端值測試。接下來預計實際打 server cmd，並結合 Chrome DevTools MCP，讓 H5 猴爺平台的自動化測試更加完善。

03 / H5 老虎機 Info 頁產生工具（Builder）

把多語系 Info 頁製作流程工具化

這是一個單檔離線可用的網頁工具，將遊戲企劃產出的 Excel 多國語言表與圖片素材，半自動轉換成符合公司規格的 7 語系 info 頁 HTML，並打包成 ZIP 交付部署。

工具定位

  • 檔案為 builder.html，瀏覽器雙擊即可使用。
  • 零安裝、零建置、不需 server。
  • 支援繁中、簡中、英文、日文、泰文、印尼文、越南文。

解決痛點

  • 不再逐語系手動貼翻譯文字。
  • 圖片可由資料夾讀取並智慧配對。
  • 既有 HTML 可匯入修改後再匯出。
  • 右側可即時預覽正式 CSS 版面。
Builder 核心流程

從 xlsx 到 7 語系 ZIP

1

匯入 xlsx，或匯入已有 HTML。

2

讀取 images 資料夾，自動分類符號圖、特殊圖騰、截圖與派彩線。

3

切換語系 tab，微調各語系內容。

4

使用即時預覽確認版面效果。

5

一鍵匯出 7 份 HTML 與用到的圖片，打包成 ZIP 後即可部署。

Builder 功能整理

功能與技術重點

核心功能

  • xlsx 匯入：支援賠率表與多國語言表雙 sheet。
  • HTML 匯入：反向解析 7 份 HTML 後回填工具。
  • 智慧圖片佔位符：如 {WILD圖}、{SC圖}、{M1圖}。
  • 動態參數：支援 {arg1} 等 URL query 參數。
  • 自我檢查：偵測中文殘留、重複路徑、未替換佔位符等問題。

技術特點

  • 單一 HTML 檔執行，瀏覽器直接開啟。
  • 使用 SheetJS 解析 xlsx，JSZip 進行打包。
  • 匯出 HTML 套用既有 paytable.css。
  • 支援 JSON 匯出 / 匯入，可中斷後下次繼續。
  • 內建版本戳，方便追蹤更新。
其他使用方法 / AI 教我怎麼協助 AI 找到 API

從飲料團購 Bot，學到理解未知系統的方法

起點只是想讓 Telegram Bot 查詢飲料團購訂單，包括團員訂單明細與 nidin 訂單狀態。但問題是：nidin 沒有公開 API 文件。

這次重點不是 AI 幫我找到 API，而是 AI 教我怎麼協助 AI 找到 API。

原本可能卡住的點

  • 沒有公開 API 文件，不知道從哪裡查資料。
  • 系統畫面看得到結果，但背後 request 與欄位未知。
  • AI 無法直接看到登入後台與 Network 資訊。

AI 教我的做法

  1. 開啟 nidin 後台並按 F12。
  2. 切到 Network，實際操作「查看訂單」。
  3. 找出相關 request，複製成 curl。
  4. 貼回給 AI 分析 API path、header、token 與 response。
其他使用方法 / 人與 AI 的分工

把「看不到的資訊」整理成 AI 可以分析的材料

我能做到的

  • 登入 nidin 後台。
  • 操作實際畫面。
  • 用 F12 看到 request。
  • 複製 curl、token、response。

AI 能做到的

  • 判斷哪個 request 是關鍵。
  • 分析 API path，例如 order/{id}/info。
  • 看懂 header 與 MC-API-Token。
  • 解讀 response 欄位，並協助寫成 Bot 功能。
AI 不一定一開始就知道答案，但它可以教我們怎麼觀察、怎麼整理資料，並把未知問題拆成它能協助分析的形式。
AI 不只是幫忙產出內容，
更重要的是協助我們整理流程、降低錯誤，
再把文件、測試、交付與日常使用方法變成能持續進化的工具。

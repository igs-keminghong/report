跟工作無關的 AI 應用

AI 教我怎麼
協助 AI 找到 API

從一個飲料團購 Telegram Bot，學到一種更實用的 AI 協作方式。

起點：只是想查飲料團購訂單

我做了一個 Telegram Bot，原本只是想幫忙處理飲料團購。

後來我想加兩個功能：

  • 查看團員訂單明細：誰訂了什麼、數量多少、有沒有備註
  • 查看 nidin 訂單狀態：目前訂單是成立、處理中，還是其他狀態
問題是：nidin 沒有公開 API 文件。

以前可能會卡住

沒有 API 文件，通常第一個反應是：

那是不是就沒辦法了？

但這次 AI 沒有直接亂猜 API。

它反而教我一件事：

你可以幫我把我看不到的資訊找出來。

AI 教我的方法

  1. 開啟 nidin 後台
  2. 按 F12
  3. 切到 Network
  4. 實際操作「查看訂單」
  5. 找出相關 request
  6. 複製成 curl
  7. 貼回給 AI 分析

我跟 AI 的分工

我能做到的

  • 登入 nidin 後台
  • 操作實際畫面
  • 用 F12 看到 request
  • 複製 curl、token、response

AI 能做到的

  • 判斷哪個 request 是關鍵
  • 分析 API path
  • 看懂 header 和 token
  • 解讀 response 欄位
  • 幫我寫成 Bot 功能

找到的關鍵

後來從 request 裡找到查詢訂單資訊的 API：

order/{id}/info

並且發現需要帶上：

MC-API-Token

有了這些資訊後，Bot 就可以開始查 nidin 訂單資料。

最後完成的功能

查看團員訂單明細

  • 團員名稱
  • 訂購品項
  • 數量
  • 備註
  • 訂單內容

查看 nidin 訂單狀態

  • 訂單目前狀態
  • 團購是否成立
  • 是否仍在處理中
  • 目前進行階段

真正有收穫的地方

這次重點不是：

AI 幫我找到 API。

而是：

AI 教我怎麼協助 AI 找到 API。

AI 不一定一開始就知道答案，但它可以教我怎麼觀察、怎麼整理資料，並把未知問題拆成它能協助分析的形式。

這跟工作有什麼關係？

雖然這次只是飲料團購，但我後來發現，這其實不是單純在找 API，而是在練習一種「理解未知系統」的方法。

對軟體工程師來說，很多工作情境也會遇到類似問題：文件不完整、流程不清楚、資料來源很多，但需要把它整理成可實作的功能。

  • 分析既有系統流程：觀察前後台操作，整理出實際資料流與關鍵步驟
  • 協助測試工具開發：把重複操作、查詢資料、比對結果的流程轉成工具
  • 整理 API response：請 AI 協助理解欄位意義、資料結構與可能狀態
  • 排查問題來源：從 request、response、log 中找出異常點
  • 競品或功能研究：在合法範圍內觀察操作流程，整理成可參考的設計方向
  • 降低溝通成本：把零散資訊整理成文件、流程圖或可交接的說明

前提是要在合法、有權限、符合公司資安與資料使用規範的情況下使用。

AI 不一定知道答案，
但它可以教你怎麼把答案找出來，
再一起把它變成可用的功能。
